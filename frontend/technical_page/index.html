Engineering Decisions

Trade-offs and Rationale

Decision 01 Data Engineering

Synthetic Data over Real PII

Chose SimPay VN (CTGAN 500k + WGAN-GP 50k)
Rejected Real Vietnamese banking PII

No access to real data. SimPay VN generates statistically faithful synthetic transactions with Vietnamese merchant names, bank codes, and seasonal patterns (Tet, Mid-Autumn).

Decision 02 Machine Learning

XGBoost + Autoencoder Ensemble

Chose Dual-branch: XGBoost + Autoencoder
Rejected Single-model approach

XGBoost handles known fraud signatures via supervised learning. Autoencoder catches zero-day anomalies via reconstruction error. Ensemble merges both with learned weighting.

Decision 03 Feature Engineering

DuckDB over Spark for Features

Chose DuckDB (columnar, in-memory)
Rejected Apache Spark (cluster overhead)

550k records fit in memory. DuckDB computes 200+ velocity features (per-account, per-device rolling windows) in under 3 seconds. Spark adds cluster overhead with no throughput benefit at this scale.

Decision 04 Explainability

SHAP over LIME for Explanations

Chose SHAP (Shapley values, theoretically grounded)
Rejected LIME (local perturbation, less stable)

Banking regulators require explainable AI. SHAP provides consistent feature attribution across similar inputs, specifically satisfying SBV (State Bank of Vietnam) requirements for algorithmic transparency.

Decision 05 Production API

FastAPI + Redis Architecture

Chose FastAPI async + Redis feature cache
Rejected Flask sync (no caching layer)

Redis caching reduces p95 latency from 85ms (cold) to 12ms (warm). Stateless API design enables horizontal scaling behind a load balancer.

Decision 06 Experimentation

SMOTE Rejected for Oversampling

Chose CTGAN (preserves joint distribution)
Rejected SMOTE (degraded precision by 4.2%)

SMOTE minority samples bled into the majority class boundary. CTGAN preserves distribution structure and generates more realistic fraud patterns.
